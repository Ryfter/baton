## Fixes

### Stale-candidate re-scoring (Slice A deferral, closed)
An `--apply` champion swap nulls every other active candidate's `win_rate_vs_champion` — and
nothing ever re-scored them, so one swap permanently removed the rest of the pool from the Pareto
front, parent selection, and challenger selection. Now the next `/baton:optimize-prompt` run
re-scores every stale active against the CURRENT champion (same minibatch evaluator) before
evolving, printing `rescored pNNN vs champion: <rate>` and returning a `rescored` result field.
Spend happens only inside explicit optimize-prompt runs — never on `/baton:go`. A no-evidence
re-score leaves the candidate honestly stale; unreadable files are skipped with a stderr note.

### Assigned-challenger verdict attribution (v1.7.0 review M2, closed)
`Complete-Run` accrued dollars to the variant recorded in `shadow.json` but judged the verdict
against whoever challenger selection would pick NOW — a mid-run evolution could misattribute one
verdict cycle. `Get-ShadowVerdict` gains `-ChallengerId`, and `Complete-Run` passes the
`challenger_id` pinned at plan time. Assigned-but-retired reads as `no-challenger` (no action,
fail-open). The CLI `--pool` report keeps the live pool's view by design.

### One-shot PROMOTE nudge (v1.7.0 review M1, closed)
A winning challenger re-emitted the "promote via --apply" event on every gated run. New additive
`promote_recommended_at` stamp (covered by `Get-PromptPool`'s DateTime re-stringify, same
`ConvertFrom-Json` trap as `created`/`retired_at`) makes the run-log nudge fire once per
candidate; `--pool`'s verdict footer now shows `(recommended <stamp>)`.
